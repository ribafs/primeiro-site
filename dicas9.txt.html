			GERENCIADORES DE JANELA E X

Qual a diferença entre o X e Gerenciador de janelas, se eu ñ me
> engano Gerenciador de janelas seria o kde ou WMaker, e o X que bicho é
> este?????
        Bem, vamos lah. Vou ser bem simplista.
        O X eh o "servidor grafico". Em Unix (e Linux), o ambiente grafico
funciona como cliente-servidor. O X, no caso, eh o servidor. Os programas
que voce abre no X (xterm, netscape, etc) sao clientes. Ou seja, eles se
"conectam" no X e sao abertos. Daih voce poder chamar programas de fora do
X e eles aparecem no X (experimento: abra o X, digite Ctrl-Alt-F2, voce
vai cair numa tela nao grafica, digite "DISPLAY :0.0" sem aspas, digite
"xterm" sem aspas, digite Ctrl-Alt-F7 pra voltar pro X e veja lah aparecer
o xterm que voce chamou do console). Alias, eh possivel fazer isso mesmo
de outra maquina, alem de "requisitar" programas para o seu X (jah abri
muito Emacs de outras maquinas no meu X).
        Bem, entao, resumindo, o X eh o servidor grafico.
        Soh que, por decisao de design do pessoal que desenvolveu o X, ele
nao se preocupa com a aparencia, com menus, com nada disso (experimente
digitar no console soh "X", ao inves de "startx"; daih pode repetir a
experiencia acima).
        Daih entram os gerenciadores de janelas (window managers). Eles
sao os responsaveis por definir a aparencia e comportamento das janelas,
mouse etc no seu X.
        O KDE e o Gnome sao um pouco diferentes disso. Nao sao window
managers (wm), mas ambientes graficos integrados. O Gnome ateh aceita que
voce rode um wm ao mesmo tempo em que ele eh rodado (o KDE, que eu saiba, 
nao). Eles se preocupam em integrar os objetos que estao no seu X. Isso
possibilita, por exemplo, "Drag and drop".

        Complementando, fiz algo um pouquinho mais elaborado, pra poder
usar a impressora tb. Carrego o modulo da impressora no boot.
        Passo o tipo de filesystem do zip como parametro pro script. Por
default, ele usa ext2.
        Vendo agora, fiz algumas mudancazinhas, pode ter algum erro. 

#!/bin/bash
# rbp@pobox.com

if [ "$1" == "-" ]
    then
    umount /zipdrive
    PPA=`lsmod | grep ppa | cut -f 1`
    if [ PPA == "ppa" ]
           then
           # Tira ppa"
           rmmod ppa
    fi
    LP=`lsmod | grep lp | cut -f 1`
    if [ $LP != "lp" ]
           then
           # Insere impressora
           modprobe lp
    fi
else
    if [ "$1" == "" ]
        $TYPE='ext2'
    else
        $TYPE=$1
    fi
    # Checa se o modulo de impressora estah carregado e o remove,
    # se estiver
    LP=`lsmod | grep lp | cut -f 1`
    if [ $LP == "lp" ]
           then
           # Tira impressora
           rmmod lp
    fi
    PPA=`lsmod | grep ppa | cut -f 1`
    if [ PPA != "ppa" ]
           then
           # Insere ppa
           modprobe ppa
    fi
    mount -t $TYPE /dev/sda4 /zipdrive
fi

		ACENTUAÇÃO EM CLIENTES DE E-MAIL

Acentuação em E-Mail
Novidades

   * 21/03/1999: Criado.

Muita gente diz que no Linux você não consegue mandar mails com acentos.
Isso não é verdade.

O que você precisa é de um programa que suporte letras acentuadas, e
configurá-lo corretamente. E isso não é uma exclusividade do Linux: é a
mesma coisa com aplicativos do Windows ou de qualquer outro Sistema
Operacional. Se o suporte for falho, você vai ter problemas em enviar
mensagens acentuadas.

Para exemplificar: a versão em japonês do InternetMail e do InternetNews da
Microsoft usava, como padrão, caracteres não compatíveis com a Internet.
Você enviava uma mensagem, e apenas pessoas que usavam o InternetMail
conseguiam ler a sua mensagem. Não sei se isso foi corrigido no Outlook, mas
a imagem da Microsoft ficou bastante comprometida entre os usuários mais
experientes.

1. Conjunto de Caracteres (charsets)

Antes de mais nada, devemos saber o que são os charsets.

1.1. Charsets

Quando o computador foi criado nos Estados Unidos, eles criaram uma tabela
de caracteres com o alfabeto, números e outros sinais, como parênteses,
exclamação etc. Para isso, 128 caracteres eram o suficiente, e portanto eles
criaram um charset de 7 bits chamado ASCII.

Porém, no mundo existem vários países que usam letras acentuadas, e 128
caracteres se tornaram insuficientes. Assim, foram criados charsets de 8
bits (256 caracteres). Entretanto, mesmo 8 bits são insuficientes para
guardar todas as letras acentuadas e especiais que são usadas nas línguas
européias, sem contar os alfabetos cirílico (Rússia) e grego, que são
totalmente diferentes dos alfabeto que nós usamos.

Assim, foram criados vários charsets de 8 bits, e normalizados pela ISO
(International Standardation Organization). Todos eles são compatíveis com o
ASCII nos primeiros 128 caracteres, diferenciando-se entre si nos últimos
128 caracteres.

Se você quiser ler um texto em russo ou em sueco, basta mudar o charset (se
o programa suportar, é claro). Um programa corretamente implementado não
deve ter dificuldade em mostrar várias línguas.

Para alguns países da Ásia, onde se usam ideogramas, o charset é de 16 bits.
Nesse caso, os aplicativos devem ser modificados para poderem trabalhar com
esses caracteres.

Devemos nos lembrar também que, dentro de um mesmo país, pode haver mais de
um charset sendo usado. Por exemplo, aqui no Brasil usamos o ISO-8859-1 no
Linux, e o IBM 850 no DOS, que são incompatíveis. No Japão, existem 3
charsets diferentes (JIS usado na Internet, shift-JIS usado no
DOS/WIN/MacOS; EUC usado no UNIX).

Os charsets mais usados aqui no Brasil são:

   * ASCII: charset de 7 bits, sem os caracteres acentuados. É o padrão nos
     Estados Unidos.
   * ISO-8859-1: charset e 8 bits, com caracteres acentuados usados nos
     países da Europa Ocidental. O português é incluído nesse charset.
   * IBM 850: charset de 8 bits, usado no DOS. Muita gente já deve ter
     percebido que os caracteres acentuados escritos no DOS ficam "errados"
     no Windows9x ou no Linux.

1.2. Limitações dos charsets atuais

O problema é: você só consegue usar um charset por vez. Portanto, você não
consegue ler um texto que tenha letras acentuadas em português e alfabeto
cirílico (charset ISO-8859-5) ao mesmo tempo. Outro exemplo: você manda um
texto em português para um amigo que mora na finlândia (charset ISO 8859-4).
Ele pode ter dificuldades em ler o texto se estiver usando um sistema
operacional finlandês que não permita mudar de charset.

Para solucionar esses problemas, um novo código de 16 bits (65536
caracteres), chamado de Unicode, pretende padronizar todos esses charsets em
um. Isto é, você poderá ler caracteres europeus ou asiáticos sem ter que
mudar os charsets, e ao mesmo tempo!

A princípio, os arquivos somente com alfabeto ficariam maiores, pois usariam
2 bytes por caracter ao invés de 1 byte. Para evitar isso, foi criada uma
forma alternativa para o encoding do Unicode (UTF-8):

  1. UTF-16: os caracteres são definidos em 16 bits, com possibilidade de
     expansão para mais de 1 milhão de caracteres.
  2. UTF-8: usa caracteres de bits variáveis. Um caractere alfabético usaria
     apenas 7 ou 8 bits, e caracteres asiáticos usariam 16 bits.

Mas ainda persiste um problema: japoneses, chineses e coreanos (que seriam
os mais beneficiados pelo Unicode) não gostaram muito da maneira com que a
tabela de caracteres foi definida. Eu não sei os detalhes, mas dizem que
houve uma simplificação exagerada prejudicando a compatibilidade com as
normas atualmente utilizados.

2. Particularidades do e-mail

A Internet foi criado nos Estados Unidos, quando a largura de banda era
estreita. Por isso, os servidores só deixavam passar caracteres de 7 bits, o
que é o suficiente para o ASCII.

Quando a Internet passou a ser usado no mundo inteiro, foi preciso modificar
o padrão para poder transmitir caracteres de 8 bits (Europa) e de 16 bits
(Ásia). Para isso, foram criados o "quoted-printable" para os caracteres de
8 bits e outras implementações locais para os países que usam caracteres de
16 bits (por exemplo, o JIS no Japão).

Esses métodos usam apenas 7 bits de informação, mas usam os
escape-characters para poderem transmitir caracteres extras. Por exemplo,
"São Paulo" se torna "S=E3o Paulo" quando transmitido no modo
"quoted-printable". O grande problema desse método é que apenas mailers que
o suporta consegue mostrar os caracteres corretamente. Um editor de texto,
por exemplo, mostra os códigos, e não os caracteres acentuados.

Mais recentemente, os servidores mail passaram a aceitar caracteres de 8
bits. Portanto, ficou muito mais fácil para os mailers trabalharem com
caracteres de 8 bits. Mas alguns servidores (especialmente nos Estados
Unidos e na Ásia) continuam a transmitir apenas em 7 bits. Portanto, se você
usar um charset de 8 bits a sua mensagem _pode_ ficar com os caracteres
trocados (o "ç" se torna "g" e assim por diante).

O Thiago Macieira me informou que o Sendmail é capaz, inclusive, de fazer
conversões de 8 bits para 7 bits e vice-e-versa.

3. Programas de E-mail problemáticos

O e-mail pode ser mandado de um país para outro com grande facilidade.
Porém, o charset usado na mensagem pode ser diferente do usado normalmente
pelo receptor. Por exemplo, o caractere "õ" do charset ISO-8859-1 (que
usamos aqui no Brasil) sairia como "o com duplo agudo" em um país que usa o
charset ISO-8859-2.

Para contornar isso, basta indicar o charset usado no conteúdo da mensagem
no cabeçalho (header), e o mailer interpretá-lo corretamente.

Temos aqui como exemplo o cabeçalho de algumas mensagens:

   * Transmitido em 8 bits:

     Content-Type: TEXT/plain; charset=iso-8859-1
     Content-Transfer-Encoding: 8BIT

   * Transmitido em 7 bits:

     Content-Type: Text/Plain; charset=iso-8859-1
     Content-Transfer-Encoding: quoted-printable

   * Mensagem em japonês, transmitido em 7 bits:

     Content-Type: Text/Plain; charset=iso-2022-jp
     Content-Transfer-Encoding: 7bit

O problema é que alguns programas de e-mail ignoram esse procedimento.
Alguns mandam sem especificar o charset, e outros simplesmente mandam
caraceres acentuados com charset=ASCII! Alguns exemplos:

   * Kmail (falta o charset):

     Content-Type: text/plain
     Content-Transfer-Encoding: quoted-printable

   * Yahoo (charset errado):

     Content-Type: text/plain; charset=us-ascii

   * Hotmail (falta o charset, não tem o encoding):

     Content-Type: text/plain

Muitos programas de e-mail são tolerantes, e (provavelmente usando o charset
padrão) mostram as letras acentuadas corretamente mesmo sem esses dados. Mas
isso porque é uma mensagem em português mandada para alguém que mora no
Brasil, pois se fosse em outra língua, provavelmente daria problemas.

Outros são rigorosos, e acabam mostrando caracteres errados. Por exemplo, o
mailer que eu uso consegue mostrar praticamente todos os charsets existentes
automaticamente (se tiver as fontes). Mas ele precisa do charset para isso,
e mensagens enviadas por esses mailers ficam corrompidas.

4. Como configurar o seu programa de mail

Basicamente, você deve configurar o seu programa para transmitir os dados
usando o charset ISO-8859-1. A transmissão em si pode ser feito em 7 bits
(quoted-printable) ou em 8 bits, mas nesse caso existe a possibilidade
(remota, hoje em dia) de você ter as letras acentuadas transmitidas
erradamente.

				BACKUP DO KERNEL

é simples, copie o kernel que está funcionado com outro nome:
cp /boot/vmlinuz-seu-kernel-atual /boot/vmlinuz-seu-kernel-de-panico

e coloque a entrada para ele no /etc/lilo.conf

[...]
image = /boot/vmlinuz-seu-kernel-atual
  label = linux --------------------------> label padrão
  root = /dev/hdb1
image = /boot/vmlinuz-seu-kernel-de-panico
  label = linux1 -------------------------> label que você quiser
  root = /dev/hdb1
[...]

finalmente é só dar o velho lilo -v e pronto.

Dai quando voce for atualizar seu kernel, sempre tera o velho
vmlinuz-seu-kernel-de-panico pronto para ser acionado no boot com
Lilo linux1

Aqui eu devo ter uns 3 ou 4 kernels :-)

			BACKUP DE CONFIGURAÇÕES

Outra dica legal para não perder os arquivos de configuração é colocar
todos, ou ao menos os mais importantes e que você levou mais tempo para
configurar em uma partição separada de backup e fazer um link para os
diretórios de origem:

mv /etc/sudoers /backup/etc/sudoers
ln -s /backup/etc/sudoers /etc/sudoers

Assim você terá em uma partiçào separada todos seus arquivos de
configuração. No caso de algum pânico é so linkar tudo novamente (podes
até fazer um scrip para isso :-)

Block devices  --->
       [ ]    Use old disk-only driver on primary interface
      <*>    Include IDE/ATA-2 DISK support
      < >    Include IDE/ATAPI CDROM support
      < >    Include IDE/ATAPI TAPE support
      < >    Include IDE/ATAPI FLOPPY support

      [*]    CMD640 chipset bugfix/support

emulação SCSI.

Em seguida selecione:
    SCSI support  --->
      <*> SCSI support
      --- SCSI support type (disk, tape, CD-ROM)
      < > SCSI disk support
      < > SCSI tape support
       SCSI CD-ROM support
        [ ]   Enable vendor-specific extensions (for SCSI CDROM)
       SCSI generic support
       --- Some SCSI devices (e.g. CD jukebox) support multiple LUNs
        [ ] Probe all LUNs on each SCSI device
       [*] Verbose SCSI error reporting (kernel size +=12K)

Nesta opção vc deverá ativar o suporte a CD-ROM SCSI  e SCSI generico, note

NFS Filesystem support - Para conexão local de micros Unix: SLIP, PLIP, PPP

Você está se referindo à saída de um ls com cores e com a opção -F. As cores
podem variar de distribuição para distribuição, mas o -F não. O geral é assim:
/ (azul)    diretório
@ (ciano)    link simbólico
@ (vermelho)    link simbólico quebrado
* (verde)    executável
= (rosa)    socket
| (marrom)    pipe
(amarelo)    dispositivo

Os executáveis são identificados pelos seus atributos e são exibidos com a cor
verde e seguidos do asterisco (*). Para rodá-los, escreva seu nome. Se não
incluir o nome do diretório, só irá encontrar os executáveis no $PATH.

			LIMITAÇÃO DE PARTIÇÕES

> O problema e que so existe devices ate o hd?16, ele quer criar o hd?17 mas
> o MAKEDEV diz:
> ./MAKEDEV: don't know how to make device "hda17"
>
> entao:
>  Como se cria o /dev/hda17 ???

Se o MAKEDEV não sabe criar, você tem que usar o mknod na marra.

De acordo com o /usr/src/linux/Documentation/devices.txt, as partições dos HDs
nas IDEs 0 e 1 em PCs são da seguinte forma:
major: 3
minor: drive * 64 + partição
Ou seja, hda = 0 * 64 + 0 = 0
hda1 = 0 * 64 + 1 = 1
hda17 = 0 * 64 + 17 = 17
hdb = 1 * 64 + 0 = 64
hdb1 = 1 * 64 + 1 = 65
etc.

daí, use o mknod. Não lembro bem sua sintaxe (leia-se: man mknod), mas seria
algo do tipo:
mknod b 3 17 /dev/hda17
(b de block, em oposição a c de character)

E lembre-se de acertar as permissões:
chown root.disk /dev/hda17
chmod 640 /dev/hda17
(ou como preferir)

>  exite limitacao quando o numero de particoes logicas??

Hmm, não sei. Mas, se existir, aposto que é uma potência de 2 :-). Embora você
tenha acabado de cruzar o número 16, você tem somente 12 partições lógicas.
Portanto, se o limite for de 16 partições lógicas, você poderia ir até o
hda20.

Se você tiver um sistema INTEIRO dentro de outro diretório, como por
exemplo uma partição montada, utilize:
rpm --root  -ivh pacote.rpm

Se você quiser só instalar em um diretório diferente mas no mesmo
sistema, eu conheço uma solução. Não testei com outras versões, mas
funciona com o RPM 3.0 com pacotes do RPM 3.0:
rpm --prefix  -ivh pacote.rpm

rpm --relocate = -ivh

Incompatibilities between LILO >= 20 and earlier versions
=========================================================

Special chain loaders
---------------------

The functionality of the chain loaders any_b.b and any_d.b
has been generalized and integrated into chain.b. any_b.b
and any_d.b are therefore no longer supported. Their
behaviour can be emulated as follows:

   other = foo
     loader = /boot/any_b.b

becomes

   other = foo
     map-drive = 0
       to = 1
     map-drive = 1
       to = 0

   other = foo
     loader = /boot/any_d.b

becomes

   other = foo
     map-drive = 0x80
       to = 0x81

       to = 0x80
