Com a ajuda do Cavassin, consegui inserir o cedilha no WordPerfect.
Minha solução (devem haver outras) foi:
1) Preferi editar o teclado do XWP80 ao invés de criar outro. 
2) Chame o WordPerfect como root (deve pedir a licença, mas não ligue).
Se não chamar como root, você vai ter que criar outro teclado (isto
vai ficar mais claro adiante). 
3) Menu Preferences -> Keyboard  
4) Aqui e que aparece a opção de ou editar o XWP8 ou criar outro
teclado. Eu preferir Edit _XWP8_ 
5) Aparece a janela do  Keyboard Editor. 
6) Procure o c + Alt (selecione-o). Selecione "Play a Keyboard script"
7) Clique no quadrado em branco. Digite Crtl-W 
8) Aparece o menu Symbols. No Set, escolha Multinacional. 
9) Na terceira linha está o cedilha minúsculo. Selecione-o. O Indicador
Number deve ir para 1,39. Clique OK.
10) Deve ter aparecido um cedilha minúsculo no quadrado, outrora
totalmente branco. 
11) Clique Assign Script. 
12) Repita os passos de 6 a 11 para o Ç . 
13) Clique OK. 
14) O cedilha minúsculo vai estar no Alt-c e o maiúsculo no
Alt-Shift-C. 
 
P.S.: você pode alterar o teclado default sem chamar o xwp como root.
Basta alterar as permissões dos arquivos _XWP8_.wgk no diretório onde
você instalou o WP. Ou ainda alterar todas as permissões daquele
diretório (só que você vai ter que ser root para fazer isto :)

			SOLUÇÃO PARA DUAS RESOLUÇÕES NO X

Tenho um monitor Samsung SyncMaster 700s (17"). Minha placa de vídeo é
uma Trident
TGUI 9680 que o sistema (XConfigurator) identificou como TG9660. Ela tem
2MB de
RAM (acho que todas deste modelo vêm só com isso).

Uso dois modos: 1280x1024 com 8bpp e 1024x768 com 16bpp. 
Para isso editei o arquivo XF86Config, retirando todos os modelines
exceto
os 1024x768 @ 70 Hz e 1280x1024 @ 61 Hz (no caso específico de meu
sistema).

Desta maneira (se não esqueci nada) o X nasce com 1280x1024 e 8bpp. Para
usar
o Netscape com uma resolução melhorzinha, abro outra sessão do X:

$ startx -- :1 -bpp16 &

e como é 16 bpp, ela abre com 1024x768.

Conforme preciso, vou alternando entre sessões com Ctrl_Alt_F7 e
Ctrl_Alt_F8.

A vantagem é que não preciso mais ficar teclando Ctrl_Alt_+ o tempo
todo,
principalmente quando entrava no sistema (no boot) e em algum modo
estranho
(tela virtual, baixa resolução) e tinha que ficar caçando com Ctrl_Alt_+
até encontrar um modo decente.

	CALCULANDO A MEMÓRIA NECESSARIA PARA UMA PLACA DE VÍDEO

Para conseguirmos 16 cores por pixel de imagem precisamos de 2 elevado à 
quarta potência, ou seja, precisamos de 4 bits. Para 256 cores, vamos ter 
que usar 8 bits.
Para 65.536 cores (high color) necessitamos de 16 bits e para 16,7
milhões de cores, usaremos 24 bits. Existem ainda placas que trabalham com
32 bits de cor. Assim, o cálculo para memória de vídeo é feito da seguinte 
forma: memória de vídeo necessária (em bits) = número de colunas x número de
linhas x número de bits para as cores. Por exemplo, se você quiser trabalhar
na resolução de 800 x 600 (800 colunas e 600 linhas) com 256 cores (8 bits 
para cor), sua placa de vídeo terá que possuir a seguinte memória: 800x600x8
bits, que é igual a 3840000 bits, ou ainda, 480.000 bytes (aproximadamente 
0,5 MB). Ou seja, uma placa com 1 MB vai trabalhar tranqüilamente nesta
configuração. Se você quiser trabalhar com 1280 colunas, 1024 linhas e 32
bits de cor vai precisar de 1280x1024x32 = 41943040 bits = 5242880 bytes = 
5 MB de memória para placa de vídeo. Como as placas normalmente só trabalham
com potências de 2 para memória, você vai ter que usar uma placa com 8 MB. 
Os aplicativos vão precisar de mais memória de vídeo se eles usarem uma 
maior resolução ou número de cores. Em geral, só os aplicativos gráficos 
necessitam de maiores resoluções e cores.

			USANDO FONTES TRUE TYPE NO LINUX


1- Copie as suas fontes .TTF para o dir /usr/X11R6/lib/X11/fonts/TrueType. Para
usar as fontes tanto no Linux quanto no rWin, apague o dir TrueType e crie um
link simbolico para o dir de fontes no windows (ln -s /dos/windows/fonts
TrueType).
2- Entre no dir onde estao as TTF e rode o ttfkfdir (se vc n tiver, pega em    
http://www.darmstadt.gmd.de/~pommnitz/xfsft.html)
3- Pegue o arquivo que ele criar (fonts.scale) e copie como fonts.dir (nao
mova, copie). Se voce estiver na particao do win, nao ha como criar symlink,
entao o negocio e copiar mesmo.
4-coloque a linha "xset +fp /usr/X11R6/lib/X11/fonts/TrueType" (sem aspas) em
algum arquivo de inicializacao do X (aqui eu coloquei no ~/.Xsession).
Da proxima vez que voce entrar no X, ja vai poder usar TTF. Mas se voce esta se
mordendo de ansiedade pra ver ttf no X, digite a linha xset...(mesma q acima).

Em caso de duvidas: 
/usr/X11R6/lib/X11/doc/TrueType/
http://www.freetype.org
http://www.darmstadt.gmd.de/~pommnitz/xfsft.html
man 1 xset

Depois de esquentar os neuronios perder algumas centenas de fios de cabelo,
consegui configurar a resolução 800x600. Ai vai, para aqueles que "ainda"
tem um monitor SyncMaster 3, configure no arquivo XF86Config:

   Identifier  "SyncMaster 3"
   VendorName  "Samsung"
   ModelName   "3"
   BandWidth   45.0
   HorizSync   30-40
   VertRefresh 50-87
   Modeline    "640x480"  25  640  672  768  800  480  490  492  509
   ModeLine    "800x600"  36  800  824  896 1024  600  601  603
 625 -hsync -vsync
   Modeline    "1024x768" 45 1024 1040 1210 1267  768  777  785  817
Interlace
EndSection
-------
Não sei qual sua distribuição e nem qual o monitor, então vou tentar ser
mais generico possivel.

Primeiro, tenha certeza de que sua placa de vídeo esta configurada
corretamente. Depois, observe a sessão "Monitor" do seu arquivo. Tentarei
descrever o que fiz no meu.

Procure no manual do monitor os valores de frequencia horizontal em kHz,
vertical em Hz e dot clock (ou dot frequency, ou pixel frequency, algo
assim) em Mhz, tipo de polarização do sinal de sincronismo de vídeo e a
largura de faixa (ou Bandwidth) em Mhz(confusão, né) PARA CADA MODO DE
RESOLUCAO!!!.

Deve existir uma tabela com este valores, por exemplo, no meu caso,
encontrei para 800x600:

fH=35.45 Khz
fV=46 Hz
dot freq=36 Mhz   <--- este é o mais importante!
pol horiz=+/-     <--- positivo ou negativo (tanto faz, teoricamente)
pol verti=+/-

Como ficou meu XF86Config:

Section "Monitor"
   Identifier  "SyncMaster 3"
   VendorName  "Samsung"
   ModelName   "3"
   BandWidth   45.0
   HorizSync   30-40
   VertRefresh 50-87
   Modeline    "640x480"  25  640  672  768  800  480  490  492  509
   ModeLine    "800x600"  36  800  824  896 1024  600  601  603
 625 -hsync -vsync
   Modeline    "1024x768" 45 1024 1040 1210 1267  768  777  785  817
Interlace
EndSection

traduzindo:

Identifier, VendorName, ModelName: dados do fabricante (Identifier tem que
bater na sessao Display)

Bandwidth: Largura de faixa do monitor (MHz)

HorizSync: Faixa que engloba todos os valores encontrados no manual. Vc pode
especificar os valores discretos, separados por virgula, que "deveria"
funciona. No meu caso não funcionou pois dependia do que a placa de video
"poderia" oferecer.

VertRefresh: Idem

Modeline: por exemplo, linha "800x600", este é o modo (tem que bater na
sessao display),
36 é o dot frequency (MHz), os valores 800 ate 625 não alterei, ou seja,
copiei de uma linha já existente, -hsync e -vsync especificam polarizações
negativas para sincronismo horizontal e vertical - optei por negativa pois a
positiva não funcionou, mesmo o manual informando que poderia ser
utilizada - verifique o seu.

Voce pode tentar também "man XF86Config" que explica todos estes parametros

Agora é só queimar uns fios de cabelo...

			    USO DO GHOSTWRITER

O ghostwriter é um programa para abrir e imprimir arquivos gráficos 
tipo PostScript (tipo .ps). Este é um padrão criado pela Adobe para
arquivos gráficos, que se baseia em construção vetorial de objetos:
por exemplo, se quero uma reta, digo que ela começa no ponto (x1,y1)
e acaba no ponto (x2,y2), com espessura de l, cor azul marinho etc.
Comparando-se com padrões como o bitmap do DOS, que tem uma matriz
de pontos preenchida com pontos de cores diferentes, o PostScript
tem duas grandes vantagens:
- o código necessário para se gerar uma figura fica extremamente
  compacto;
- pode-se dar zoom (dependendo do aplicativo) sem perda de resolução
  até os limites da própria definição no arquivo.
- a vantagem anterior torna este padrão extremamente atraente para
  desktop publishing, artes gráficas em geral.

O padrão Acrobat (.pdf) também da Adobe tem várias das características
do PostScript, mas com maior portabilidade, devido à bilbioteca
restrita e padronizada.

O PostScript tem várias desvantagens:
- é interpretado, ou seja, o leitor de arquivo .ps deve ler e entender
  o arquivo, inclusive os tipos de letras etc. que são objeto de
  padrões (tipo I, II etc.). Se este leitor não tiver aquela
  biblioteca específica, nada feito;
- esta desvantagem torna este padrão difícil de ser implementado
  em aplicações "em tempo real" (joguinhos etc.);
- impressoras Post-Script têm processadores internos que lêem o
  arquivo e interpretam-no, gerando a saída "bitmapeada" conforme
  a resolução que o freguês quer.
- no mundo DOS/Windows este padrão é virtualmente desconhecido.
  A Microsoft seguiu outros caminhos, como os fontes TrueType etc.,
  com o intuito principal de não pagar royalties à Adobe nem a
  ninguém.

O Ghostwriter é um aplicativo gratuito (GPL) que lê e imprime
Post-Script mesmo em impressoras não PostScript. Ele possui
uma interface gráfica chamada Ghostview, que mostra arquivos
.ps e .pdf, e os "imprime" para impressoras que não podem 
interpreta-los (quase todas ;-)).

Para chamar o Ghostview na linha de comando: 

[meu computador]$gv

Para chamar o Ghostwriter na linha de comando: 

[meu computador]$gs

(não faça isso ;-). Você não saberá usa-lo assim. Use o 
Ghostview).

Mas ele está instalado no menu de opções em distribuições como
o RedHat.

Ah, ia me esquecendo: o PostScript é um padrão muito popular
no mundo UNIX.

Quaisquer incoerências e incorreções, por favor escrevam-me.

		CORTAR AS DUAS ÚLTIMAS LINHAS DE UM ARQUIVO

 como eu posso fazer pra eliminar as duas ultimas linhas de um arquivo texto, de
> dentro de um shell script?

Utilize uma combinacao do awk com o wc. Assim:

file=/meu/arquivo/a/ser/cortado
total=`wc -l $file|awk '{ print $1; }'`
awk 'NR <= $total - 2' < $file

				LILO

Abaixo transcrevo uma adaptacao do
capitulo sobre LILO do livro "Using Linux" da QUE, sobre o que eh e como funciona o LILO. 
Desculpem a extensao da mensagem, mas acredito que as informacoes abaixo serao de grande valia
para muita gente. 

O programa LILO (LInux LOader) eh o carregador padrao do Linux. Esse programa eh a mais
flexivel das rotinas de partida do Linux. O LILO pode ser configurado para residir no registro
principal de inicializacao do seu HD ou no setor de inicializacao (ou de partida) de uma particao de
um HD. Voce tera de tomar essa decisao ao instalar o Linux; dependendo do seu sistema, uma ou
outra funcionara melhor para voce. 
Voce nao entendeu nada disso? Nao se preocupe... Os modernos PCs incializam quase da mesma
forma que um PC fabricado em 1982. Entretanto, o hardware mudou tanto que o processo de
partida requer que o sistema operacional e a rotina de partida para o sistema operacional estejam
conscientes dos varios rodeios que forma implementados nos PCs para suportarem itens tais como
HD maiores. 
O procedimento de partida para cada PC eh relativamente direto quando visualizados a distancia.
Ele funciona da seguinte forma: 

1. Quando o PC eh ligado, o processador inicia a execucao do codigo na BIOS. Esse codigo
dimensiona a memoria, testa o processador e o hardware e inicializa o hardware para um estado
conhecido. 
2. A BIOS entao procura pela unidade de disco de partida (geralmente um disquete ou um HD); se
eh um HD, a BIOS le o registro principal de partida e o executa. Esse registro principal de partida
(geralmente instalado pelo DOS ou Win95/98) procura pela primeira particao ativa e, depois, tenta
carregar esse registro de partida. 
3. O registro de partida da particao contem instrucoes sobre como carregar a rotina de partida. As
limitacoes de tamanho estao em efeito aqui, portanto a rotina de partida deve ser bem pequena. O
DOS em si eh pequeno o suficiente para se encaixar ai, assim, para um sistema DOS, o processo de
partida eh finalizado. Outros terao um programa aqui que carrega o kernel do sistema operacional e
o inicializa, como o LILO que carrega o kernel do Linux, descapacta-o e, depois, inicializa-o. A
memoria que a rotina de partida usa eh entao, em geral, reclamada pelo sistema operacional que foi
inicializado. 

As opcoes do LILO sao definidas e configuradas enquanto o Lunix esta em execucao, o que
significa que se o LILO nao inicializa o Linux, voce deve usar os seus discos de inicializacao de
emergencia para acessar o disco rigido do sistema e, depois, configurar o LILO. Talvez voce tenha
de experimentar para determinar se o LILO funciona melhor no registro principal de partida do seu
disco rigido ou no setor de inicializacao da sua particao primaria. 
O LILO le a sua configuracao a partir de um arquivo no diretorio /etc do sistema chamado lilo.conf.
Esse arquivo informa ao LILO o sistema operacional para o qual ele deve configurar-se e onde
instalar a si mesmo. A sua parte basica eh o arquivo lilo.conf, o qual informa ao LILO para inicializar
somente um sistema operacional, como abaixo: 

boot=/dev/hda4 
map=/boot/map 
install=/boot/boot.b 
prompt 
timeout=50 
image=/boot/vmlinuz-2.036-0.7 
label=linux 
root=/dev/hda4 
read-only 

Esse arquivo informa ao LILO para posicionar-se no setor de inicializacao da particao /dev/hda4,
para esperar por cinco segundos pela entrada do usuario no prompt LILO: e para inicializar o kernel
vmlinuz-2.036-0.7 no diretorio /boot da particao /dev/hda4. O LILO exibe um prompt e espera por
cinco segundos por uma entrada antes de inicializar a primeira imagem do arquivo lilo.conf. Observe
que o valor de tempo de espera para cinco segundos eh 50 - o LILO mede o tempo em
incrementos de 0,10 segundos.
